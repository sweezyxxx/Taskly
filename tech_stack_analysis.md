# Материалы для защиты проекта (Презентация)

## 0. Цель проекта и Задачи (Goals)
**Цель:** Разработка отказоустойчивого кроссплатформенного приложения для управления задачами с возможностью работы без интернета (Offline-First) и мгновенной синхронизацией данных через облако.

**Задачи:**
1. Спроектировать чистую архитектуру (Clean Architecture).
2. Реализовать локальное хранилище данных (Drift/SQLite) как единственный источник правды (Single Source of Truth).
3. Настроить двустороннюю синхронизацию с Firebase (Firestore) в фоновом режиме.
4. Разработать интуитивный UX/UI с адаптацией под разные размеры экранов и состояний (загрузки, ошибки, отсутствие сети).
5. Реализовать авторизацию и изоляцию данных между аккаунтами пользователей.

## 0.5 Анализ аналогов
- **Todoist:** Мощный, но часть функционала (напоминания, статистика) доступна только в платной версии. Наша система Statistics встроена базово.
- **Any.do:** Перегруженный UI, навязчивый премиум. У Taskly упор на минимализм и фокус на текущих задачах.
- **Microsoft To Do:** Экосистемно-зависимый (требует Microsoft аккаунт), в Taskly используется стандартная почта/пароль с перспективой добавления Google Sign-in.

# Анализ технологий (Для презентации)


Ниже представлен текстовый драфт для вашего слайда «Сравнение и выбор технологий», где обосновывается архитектура Taskly.

## 1. Стейт-менеджемент: BLoC vs Provider / Riverpod
- **Provider**: Слишком простой для сложной бизнес-логики с кэшированием.
- **Riverpod**: Гибкий, но требует глубокого понимания графа зависимостей и имеет меньше стандартов структурирования для больших команд, а также нарушает инкапсуляцию, если не следовать строгим правилам.
- **✅ Выбор - BLoC**: Идеально ложится на Clean Architecture. Четко разделяет Event (что случилось) и State (что стало). Строгая архитектура и предсказуемость состояний (Loading, Loaded, Error). Легко покрывается юнит-тестами.

## 2. Локальная БД: Drift vs Sqflite vs Hive
- **Sqflite**: Писать сырые SQL-запросы долго, нет type safety.
- **Hive**: Быстрая NoSQL, но плохо подходит для сложных реляционных связей (фильтрации по статусам/приоритетам).
- **✅ Выбор - Drift (ex. Moor)**: Мощная ORM поверх SQLite. Полная строгая типизация (Type-Safe Builder), встроенная реактивность (позволяет возвращать `Stream`, который сам обновляет UI при изменениях базы), что критически важно для принципа Offline-First.

## 3. Бэкенд и Синхронизация: Firebase Firestore vs REST API
- **REST API + WebSocket**: Требует написания собственного микросервиса (Node.js/Go/Python), покупки хостинга и построения сложной логики разрешения конфликтов при оффлайн-синхронизации (Offline Queue).
- **✅ Выбор - Firestore (Firebase)**: Готовый BaaS (Backend as a Service). Имеет встроенную оптимизацию оффлайн-режима из коробки! Если интернет пропадает, Firestore кэширует операции локально и сам незаметно push'ит их, когда сеть восстанавливается. Идеальный выбор для синхронизируемых ToDo-приложений.
